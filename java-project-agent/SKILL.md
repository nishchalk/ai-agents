# Java Project Agent

Generates a `PROJECT-AGENT.md` file that gives any AI agent authoritative,
project-specific guidance for a Java/Maven codebase.

## Onboarding — ask these questions first

Before scanning the project, gather context with the `AskQuestion` tool (or
conversationally if unavailable):

1. **Project name & one-sentence purpose** — e.g. "Payment service handling card transactions"
2. **Java version** — detect from `pom.xml` (`<java.version>` or `<release>` tag) if available; confirm with user
3. **Primary frameworks** — Spring Boot, Quarkus, Micronaut, plain Jakarta EE, etc.
4. **Module layout** — single-module monolith, multi-module Maven reactor, or microservices?
5. **Test strategy** — unit only, integration tests, contract tests (Pact), E2E?
6. **Code style enforced** — Checkstyle, Spotless, Google Style, custom?
7. **CI/CD pipeline** — GitHub Actions, Jenkins, GitLab CI, etc.?
8. **Key off-limits areas** — generated code, vendor directories, legacy modules not to touch?
9. **Deployment target** — Docker/K8s, bare metal, AWS Lambda, etc.?

## Project inspection steps

After gathering answers:

1. Read `pom.xml` (root) to confirm Java version, dependencies, plugins.
2. Scan `src/main/java` package structure (top two levels).
3. Check for `src/test/java` and note test framework dependencies.
4. Look for `.github/workflows/`, `Jenkinsfile`, or `.gitlab-ci.yml`.
5. Detect code-style config: `checkstyle.xml`, `.editorconfig`, `spotless` plugin config.
6. Note any `application.properties` / `application.yml` for profile awareness.

## Coding standards

### Mandatory
- Follow {style_guide} (enforced by {linter}). Run `mvn {style_check_goal}` before committing.
- All public API must have Javadoc (`@param`, `@return`, `@throws`).
- No `System.out.println` — use SLF4J (`LoggerFactory.getLogger(getClass())`).
- Checked exceptions: wrap in domain-specific unchecked exceptions; never swallow.
- Prefer immutability: `final` fields, records for DTOs (Java 16+), unmodifiable collections.
- Null safety: annotate with `@NonNull`/`@Nullable`; avoid returning `null` from public methods — use `Optional`.

### Naming conventions
| Element | Convention | Example |
|---------|-----------|---------|
| Classes | UpperCamelCase | `PaymentService` |
| Methods / fields | lowerCamelCase | `processPayment()` |
| Constants | SCREAMING_SNAKE | `MAX_RETRY_COUNT` |
| Packages | lowercase, dot-separated | `com.example.payment` |
| Test classes | `{Subject}Test` or `{Subject}IT` | `PaymentServiceTest` |

### Package architecture (if layered)
```
controller  →  service  →  repository
              ↓
            domain (no inbound deps)
```
Never let `domain` import from `controller` or `service`.

## Build & run

```bash
# Compile
mvn clean compile

# Run tests
mvn test

# Integration tests
mvn verify

# Package
mvn clean package -DskipTests

# Run (Spring Boot)
mvn spring-boot:run
```

## Testing guidelines
- Unit tests: JUnit 5 + Mockito. No Spring context unless necessary.
- Integration tests (`*IT.java`): use `@SpringBootTest` or Testcontainers.
- Aim for ≥ 80 % line coverage on `service` and `domain` packages.
- Test method naming: `methodName_stateUnderTest_expectedBehavior`.

## Java best-practice checklist to enforce

When generating or reviewing code, verify:

- [ ] No raw types (`List` → `List<T>`)
- [ ] Streams used for collections; avoid explicit loops for filter/map
- [ ] Resources closed via try-with-resources
- [ ] `equals` and `hashCode` overridden together
- [ ] `toString` implemented for domain objects
- [ ] `@Override` annotation on all overriding methods
- [ ] No mutable `static` fields (thread-safety)
- [ ] `Optional` not used as a method parameter
- [ ] Dependency injection via constructor (not field injection)
- [ ] Tests are deterministic (no `Thread.sleep`, no `new Date()` — use `Clock`)
